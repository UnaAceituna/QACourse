Тест кейс и тест - синонимы
Чек-лист - идея будущего теста, тест-кейс - уже набор конкретных данных
Тест-план - документ, где описаны нюансы, аспекты (сроки, что именно, с пом каких видов, риски, отв люди) тестирования
Тестовый сценарий - набор тест-кейсов, собр в последователность для достижения некоторой цели

ТеstData  -примеры входных данных. Делятся на позитивные и негативные,определяются в соответствии с требованиями (не могут быть придуманы просто так). 

Test-suite - набор тест-кейсов, которые объединены тем что относятся к одному тестируемому модулю, функциональности, приоритету или одному типу тестирования. СКак правило состоит более чем из одноготест кейса и выполняется всей "пачкой" в процессе тестирования.

Тест кейсы нужны для того ,чтобы найти проблемные места в требованиях, структурировать всю информацию ,ускорить регрессию, хранить и собирать информацию, отслеживать процесс тестирования

Атрибуты тест кейса
1. id  - уникальный номер
2. title - название кейса, должен быть четким и лаконичным
3. Preconditions - действия кот необходимо вполнить до прохождения тест кейса. Не может содержать ссылки на тестируемый ресурс, если у информац системы более одной среды (прод ,предпрод). Может содержать ссылки на информационные источники для этого кейса
4. steps - шаги необх для выполнения теста. Их описание должны быть чёткими, понятными, последовательными, но без лишней детализации
5. Expected results - ожидаемые после выполнения теста результаты. Должны соответствовать требованиям, либо логике. Без альтернатив, не должно быть избыточного описания
6. description  - короткое описание
7. envitonment
8. status
9. bug number
10. Postconditions - действия после прохождения тест кейса

Правила по созданию тест-кейса
-уровень детализации. Зависит от команды и продукта
-тест кейсы должны быть независмы друг от друга
-тест кейс должен содержать только ту информацию, которая важна для его прохождения
-желательно удалять данные ,созд по ходу прохождения тест кейса
-желательное не захламлять повторяющимися шагами
-тест кейсы должны не повторять ,а проверять требования
-название должно соотв тесту. Уникальное, краткое, информативное, шаги писать не надо

Если в тест кейсе не указаны конкретные входные данные, то он должен идти по положительному сценарию. Если результаты прохождения отличаются, всё равно нужн закончить тест. Закончить основную работу, и потом вернуться к смутившим вопросам
